## Целевая схема

Этот bundle фиксирует одну целевую runtime-схему и запрещает дальше смешивать
подготовительный узловой слой с пользовательским контрактом доставки модели.

### Сквозной сценарий

1. Пользователь задаёт `Model` или `ClusterModel`.
2. Контроллер публикует модель во внутренний `DMCR`.
3. Дальше delivery идёт в одном из двух режимов.

#### Режим 1. Узловой общий кэш

- На ноде есть managed local storage.
- Узловой runtime plane держит общий том и общий digest store.
- Артефакты, нужные прикладным объектам на этой ноде, подтягиваются в этот
  общий кэш.
- Прикладной объект получает модель через общий mount из узлового слоя.

Это основной целевой быстрый путь.

#### Режим 2. Запасной том прикладного объекта

- Узловой общий кэш недоступен, не готов или не может быть использован.
- Контроллер оставляет запасной путь.
- Прикладной объект получает свой том, пользовательский или managed.
- Модель раскладывается в этот том из `DMCR`.

Это запасной рабочий путь. Он нужен как защита, но не должен выдаваться за
целевую быструю схему.

### Неподвижные инварианты

- Канонический источник опубликованной модели в обоих режимах — `DMCR`.
- Прикладной код не должен знать внутренний layout узлового кэша.
- В обоих режимах прикладной код получает один и тот же стабильный набор:
  - `AI_MODELS_MODEL_PATH`
  - `AI_MODELS_MODEL_DIGEST`
  - `AI_MODELS_MODEL_FAMILY`
- Публичный `Model` / `ClusterModel` контракт не должен разрастаться
  storage-specific деталями.
- Будущий переход на более прямой локальный storage path может менять
  внутренний substrate, но не должен менять workload-facing контракт.

### Что сейчас ещё не считается завершённым

- workload-facing shared mount service ещё не считается доведённым только
  потому, что уже есть per-node runtime pod, shared PVC и prefetch;
- текущий `shared PVC + materialize-artifact` путь не должен называться
  `SharedDirect`: это всё ещё bridge-контракт поверх пользовательского тома, а
  не node-owned shared delivery;
- `node-cache-runtime` не должен воспринимать такой shared PVC bridge как
  сигнал для узлового `prefetch`, потому что workload ещё не потребляет
  per-node shared plane напрямую;
- future `SharedDirect` считается валидным только вместе с отдельной причиной
  `NodeSharedRuntimePlane`, чтобы случайные или ручные аннотации не включали
  ложный `prefetch` в узловой общий store;
- наличие узлового digest store само по себе ещё не означает, что быстрый путь
  доставки до прикладного объекта уже закончен;
- запасной путь через `materialize-artifact` остаётся текущим рабочим путём до
  landing полноценного workload-facing shared delivery.
