# Техническое задание: модуль ai-models

## Цель

Сделать модуль DKP `ai-models`, который добавляет платформе capability "единый каталог опубликованных моделей" и использует внутренний backend для хранения метаданных, экспериментов, версий и артефактов моделей.

## Продуктовый результат

В результате платформа должна получить:
- один внутренний managed backend;
- единый путь публикации моделей;
- возможность дальнейшего вывода собственного UX через `Model` и `ClusterModel`;
- возможность принимать результаты из JupyterHub, Airflow и других training/experiment контуров;
- устойчивую основу для дальнейшей интеграции с `ai-inference`.

## Архитектурная рамка

- `ai-models` — модуль платформы, а не namespaced application для команд.
- Внутренний backend — внутренний компонент модуля, а не публичный пользовательский контракт.
- Публичный контракт платформы в дальнейшем — `Model` и `ClusterModel`.
- Любой пользовательский UX поверх моделей должен строиться через DKP API, а не через прямую зависимость от внутреннего backend UI/API.

## Этапы

### Этап 1. Внутренний managed backend

Нужно получить рабочий модуль, который:
- разворачивает внутренний backend в системном namespace модуля;
- подключает backend к PostgreSQL;
- подключает backend к S3-совместимому artifact storage;
- даёт рабочий UI;
- подключается к стандартным механизмам DKP по логированию и мониторингу;
- оформлен в DKP-стиле: module values, OpenAPI, templates, Helm library, werf.

На этом этапе не нужно пытаться завершить весь каталог моделей. Главная цель — получить рабочий и проверяемый backend.

### Этап 2. Каталог моделей платформы

Нужно добавить:
- `Model`;
- `ClusterModel`;
- контроллер публикации и синхронизации с внутренним backend;
- status/conditions;
- базовый publish flow поверх внутреннего backend API.

### Этап 3. Упрочнение

Нужно добавить:
- hardening и supply-chain практики;
- controlled patching/rebasing внутреннего backend engine;
- distroless для собственного кода;
- переупаковку внутреннего backend engine только после подтверждённой пользы;
- дополнительные security checks и quality gates.

## Что важно не делать

- Не начинать с собственного registry backend с нуля.
- Не пытаться в первом же шаге сделать весь UX каталога моделей.
- Не смешивать rollout внутреннего backend с дизайном `Model` / `ClusterModel` в одном изменении.
- Не переходить к distroless или тяжёлому patching, пока нет работающего baseline.

## Требования к качеству

- Каждая задача проходит через task bundle.
- Каждый этап имеет понятный working baseline.
- Изменения укладываются в текущий этап и не тянут преждевременно будущие части решения.
- Модуль остаётся поддерживаемым и понятным без знания внутренних обсуждений.
- Любой новый крупный шаг сопровождается актуализацией docs/development.
